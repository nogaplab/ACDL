Section 1 · Overview
Overview

We thank all three reviewers for their careful reading and constructive feedback. There is broad agreement that formalizing agentic context construction addresses a genuine and important gap in the field, and that the case studies demonstrate ACDL's ability to capture meaningful structural distinctions in real-world systems. The two main concerns raised — limited empirical validation of our claims, and insufficient discussion of the language's scope and limitations — are both actionable, and we address them directly in the planned changes and per-reviewer responses below.

On empirical evaluation: we acknowledge that the paper's claims about performance consequences of context design choices are currently under-supported. Our primary goal with this work is to provide the community with a shared vocabulary for describing and comparing agentic contexts — analogous to UML for software architecture or plate notation for probabilistic models — rather than to argue for a specific context configuration. That said, reviewer C is right that even the descriptive contribution is stronger when accompanied by concrete evidence that the distinctions ACDL surfaces are practically consequential. We will expand the MINT ablation discussion with full experimental details in the revision (see Planned Changes).

On limitations and scope: the paper currently understates two important constraints. First, ACDL assumes a clear step structure; ... not yet cleanly handled. Second, multi-agent scenarios where agents share mutable state are not fully supported. We will add an explicit limitations section covering these.

Question
Why are sys and env two separate objects?
sys holds values that originate from the agent's internal state — configuration parameters, values computed during a run (e.g., generated summaries), the current time, etc. env holds values that originate from the environment the agent is interacting with — observations from a game, a user's message in a chat interface, sensor readings, and so on. The separation reflects a meaningful architectural boundary: internal bookkeeping versus external input. This distinction is often important when reasoning about an agent's information access, and we will clarify it in the paper.
Question
The DSL has ForEach but seems to lack filter, find, etc. Are list comprehensions also supported?
Filtering and find-style operations are supported via user-defined functions, which can be called within a ForEach or in an inline expression. The appendix does also show Python-style list comprehensions as syntactic sugar for common patterns.
Question
The dotted "." counter variables seem to wedge nested iteration into a flat model.
This is by design, and it reflects a fundamental tension that ACDL is trying to surface rather than hide. LLM contexts are inherently flat and linear, yet the semantic structures they encode — multi-turn conversations, nested tool call results, hierarchical observations — are often deeply nested. The dotted counter notation (e.g., i.j) makes this linearization explicit: a nested structure in the world is being flattened into a linear sequence in the context. We believe making this transformation visible is useful precisely because it is often a source of bugs and design confusion in practice. We will add a paragraph explaining this rationale in the language design section.
Question
PromptEndsHere feels like a break-ish operation; would be natural in the context of more nested loops.
The language does support break and continue for loop control. PromptEndsHere is semantically distinct: it is closer to a return — it exits the entire context construction process, not just a loop.

Concern
No evaluation comparing ACDL to writing an agentic loop in pseudocode or LaTeX. No user study.
We appreciate this concern and acknowledge it is a real limitation. That said, we note that the baseline the reviewer proposes — informal pseudocode or LaTeX — does not currently exist in practice: to our knowledge, there is no established convention for describing agentic context construction at all. ACDL's immediate contribution is to provide a starting point — a shared, structured vocabulary — that makes such descriptions possible in a consistent and machine-readable way. A user study comparing ACDL against informal alternatives is a clear and valuable direction for future work, which we will acknowledge explicitly.
Concern
The paper does not discuss out-of-scope systems or general limitations.
This is fair. We will add an explicit limitations section. The two most significant current limitations are: (1) ACDL assumes a clear, discrete step structure — systems where sub-steps are themselves prompt-driven (i.e., the context construction process is itself agentic) are not yet cleanly handled; (2) multi-agent scenarios where agents share mutable state are not fully supported in the current formalism.
Concern
The paper mentions that an LLM could implement a system from its ACDL spec, but provides no data.
This is noted as an anecdotal observation rather than a validated claim, and we agree it needs to be framed more carefully. We will reposition this as a promising direction for future work rather than an empirical finding, and remove any language that implies it has been systematically validated.
Question
Have you tried automatically generating ACDL from an existing system using an LLM?
We have experimented with this anecdotally, and our experience suggests the difficulty depends heavily on how the system is written. When a codebase cleanly separates context construction logic from execution logic, an LLM can extract a reasonable ACDL spec. When the context logic is diffuse or implied, the task becomes very hard — analogous to extracting a state chart from a large software system where the state machine is implicit. We see this as a promising and practically important research direction and will frame it as such in the paper.
Question
Have you considered tooling use cases — e.g., an automatic context manager analogous to a garbage collector?
Yes, and we agree this is one of the more exciting downstream applications of a formal context specification. The structured, data-flow-like representation in ACDL makes it possible in principle to identify dead context — portions of the context that are never referenced by subsequent steps — and to reason about caching or compression strategies. This is a compelling direction we intend to highlight as future work in the revision.

Question
Can you provide the full experimental setup for the reasoning-trace ablation?
The experiment was conducted on the MINT benchmark (excluding the alfworld dataset) using GPT4-turbo, ran once on each prompt configuration. For each task, the agent is allowed to interact with the environment up to 5 times (interaction steps include tool use (inlcluding failed attempts) and solution proposals) and up to 2 solution proposals.
