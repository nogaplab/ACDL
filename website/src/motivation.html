Why ACDL?

Understanding the need for a formal language to describe LLM context structures

The Power of Shared Languages

Clear communication of design choices is essential for reproducible and improvable engineering. Across computer science and software engineering, visual and formal languages have emerged to express system designs: UML diagrams, flowcharts, sequence diagrams, and state machines all serve as shared vocabularies that enable practitioners to describe, compare, and reason about complex systems.

In machine learning, plate notation for graphical models and the now-ubiquitous box diagrams for neural network architectures (think ResNet or Transformer diagrams) have become indispensable communication tools. Their value lies not in being executable, but in making structure visible and comparable at a glance.

The Gap in LLM Systems

Despite the rapid growth of agentic LLM systems, there is currently no standard way to precisely describe how the input to a language model is assembled and how it evolves across interaction steps. Descriptions of context construction are typically scattered across code, documentation, and ad-hoc diagrams that fail to fully capture the structure.

Even small variations in how a basic ReAct agent's context is assembled—such as whether to include reasoning steps in the history—lead to measurable differences in agent performance. These are precisely the kinds of distinctions that a descriptive language should make explicit.

What About Existing Tools?

Several languages have been proposed for structuring prompts or orchestrating LLM pipelines. However, these focus on different problems:

  • Prompt templating languages organize the content of individual prompts
  • Workflow orchestration tools execute LLM-powered pipelines
  • Agent frameworks provide APIs for building agents

None of these address the core question: how does temporally evolving system state and history map to LLM contexts across multi-turn interactions?

ACDL's Approach

ACDL is not a programming language for building agents—it's a descriptive language for specifying how context is composed, how it changes over time, and where its components originate. By providing a formal vocabulary for context structures, ACDL enables:

  • Precise documentation of how agent contexts are assembled
  • Clear communication between researchers and practitioners
  • Direct comparison of different context management strategies
  • Investigation of the design space for context construction

Once a language exists to document structural differences, the variability space becomes apparent and open for systematic investigation.
